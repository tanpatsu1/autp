# Data Model: URL Saving MVP

## Purpose

Define the smallest safe data model for the private URL-saving MVP from `docs/mvp-scope.md` and `docs/product-spec.md`. This document is planning-only: it does not create SQL migrations, change Supabase production data, add environment values, or use service role keys.

## Model Decision

Use normalized user-owned records for saved URLs, categories, and tags:

- `saved_urls`: primary user-owned URL records.
- `categories`: optional user-owned category labels; one saved URL can point to one category.
- `tags`: user-owned tag labels.
- `saved_url_tags`: join table connecting saved URLs to many tags.

Card/list display mode is presentation state, not saved URL data. For the MVP it can remain client-side local UI state. If persistent view settings are later required, add a separate `user_preferences` table after review.

## Entity Summary

| Entity | Purpose | Ownership | MVP Need |
| --- | --- | --- | --- |
| `saved_urls` | Stores one private saved URL and its user-entered metadata | `owner_id` stores the Supabase Auth user id | Required |
| `categories` | Stores a user's reusable category labels | `owner_id` stores the Supabase Auth user id | Required for clean category filtering |
| `tags` | Stores a user's reusable tag labels | `owner_id` stores the Supabase Auth user id | Required for tag search/filter |
| `saved_url_tags` | Stores many-to-many saved URL/tag links | Inherits ownership through `saved_urls` and `tags` | Required |
| `user_preferences` | Optional future display preferences such as default view mode | `owner_id` stores the Supabase Auth user id | Deferred |

## Field Model

### `saved_urls`

| Field | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| `id` | `uuid` | Yes | Generated by database | Stable saved URL identifier |
| `owner_id` | `uuid` | Yes | Current authenticated user | Supabase Auth ownership boundary; this is the app's `user_id` link |
| `url` | `text` | Yes | None | Store normalized URL string used for opening/search |
| `display_url` | `text` | No | `null` | Optional original or user-friendly display form if normalization differs |
| `title` | `text` | Yes | App-provided URL fallback | User-facing label; no metadata fetching required |
| `category_id` | `uuid` | No | `null` | Optional link to one user-owned category |
| `memo` | `text` | No | `null` | Private user note |
| `is_favorite` | `boolean` | Yes | `false` | Favorite marker |
| `created_at` | `timestamptz` | Yes | `now()` | Default sort and audit timestamp |
| `updated_at` | `timestamptz` | Yes | `now()` | Updated through trigger or app-side update path |

### `categories`

| Field | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| `id` | `uuid` | Yes | Generated by database | Stable category identifier |
| `owner_id` | `uuid` | Yes | Current authenticated user | Category is private to one user |
| `name` | `text` | Yes | None | User-entered category label |
| `normalized_name` | `text` | Yes | Lowercase/trimmed from `name` | Used for uniqueness and search |
| `created_at` | `timestamptz` | Yes | `now()` | Audit timestamp |
| `updated_at` | `timestamptz` | Yes | `now()` | Rename timestamp |

### `tags`

| Field | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| `id` | `uuid` | Yes | Generated by database | Stable tag identifier |
| `owner_id` | `uuid` | Yes | Current authenticated user | Tag is private to one user |
| `name` | `text` | Yes | None | User-entered tag label |
| `normalized_name` | `text` | Yes | Lowercase/trimmed from `name` | Used for uniqueness and search |
| `created_at` | `timestamptz` | Yes | `now()` | Audit timestamp |
| `updated_at` | `timestamptz` | Yes | `now()` | Rename timestamp |

### `saved_url_tags`

| Field | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| `saved_url_id` | `uuid` | Yes | None | References `saved_urls.id` |
| `tag_id` | `uuid` | Yes | None | References `tags.id` |
| `created_at` | `timestamptz` | Yes | `now()` | Link timestamp |

The proposed primary key is the pair `saved_url_id`, `tag_id`.

## Relationships

| Relationship | Cardinality | Rule |
| --- | --- | --- |
| User to saved URLs | One user has many saved URLs | `saved_urls.owner_id = auth.users.id` |
| User to categories | One user has many categories | `categories.owner_id = auth.users.id` |
| User to tags | One user has many tags | `tags.owner_id = auth.users.id` |
| Category to saved URLs | One category has many saved URLs | `saved_urls.category_id` is optional |
| Saved URL to tags | Many-to-many | `saved_url_tags` joins saved URLs and tags |

All user-facing records must be scoped to the authenticated user. A saved URL must not be linked to another user's category or tag.

## Data Behavior

| Behavior | Data Rule |
| --- | --- |
| Save URL | Insert one `saved_urls` row, optionally create or reuse one `categories` row, optionally create or reuse `tags`, then insert `saved_url_tags` links |
| Update URL | Update only the owner's `saved_urls` row and related category/tag links |
| Favorite | Toggle `saved_urls.is_favorite` only |
| Memo | Store private free text in `saved_urls.memo` |
| Category | Use one optional `category_id`; category labels are reusable per user |
| Tags | Use zero or more user-owned tags through `saved_url_tags` |
| Card/list toggle | Keep as UI state for MVP; do not store on saved URL rows |
| Delete URL | If later approved for CRUD completeness, delete only the owner's row and cascade only its join rows |

## Search And Listing

Default listing should fetch the authenticated user's saved URLs ordered by `updated_at desc, created_at desc`. The result shape should include:

- saved URL fields,
- optional category name,
- tag names for each saved URL,
- favorite state,
- timestamps for sorting.

MVP keyword search should match:

- `saved_urls.url`,
- `saved_urls.display_url`,
- `saved_urls.title`,
- `saved_urls.memo`,
- `categories.name`,
- `tags.name`.

Implementation can start with Supabase query composition using `ilike` for URL/title/memo and joined category/tag names, then move to a database search view or generated search column only if the first implementation becomes too complex. Advanced ranking, AI classification, metadata scraping, and recommendations stay out of scope.

## Type Expectations

Future Implementation should generate or define TypeScript shapes equivalent to:

| Type | Expected Fields |
| --- | --- |
| `SavedUrlRow` | Database row for `saved_urls` |
| `CategoryRow` | Database row for `categories` |
| `TagRow` | Database row for `tags` |
| `SavedUrlWithRelations` | Saved URL row plus optional category and tag list |
| `SavedUrlCreateInput` | URL, optional title, category name/id, tag names/ids, memo, favorite |
| `SavedUrlUpdateInput` | Editable saved URL metadata owned by current user |

When Supabase generated types are introduced, use public environment variable names only and do not commit generated types containing secrets.

## Implementation Handoff

Implementation should:

- keep the first implementation private and authenticated;
- use the public Supabase client with anon/publishable key only;
- rely on RLS for user ownership;
- avoid service role keys;
- create migrations only after review of `docs/supabase-schema.md` and `docs/rls-policy.md`;
- include tests or manual checks for cross-user access denial;
- preserve the docs-only decision that no production database is changed during `NEXT-002`.
